## Non-blocking verifier command-path Andon

Andon ID: `ANDON-20260605-V023-SUBAGENT-RERUN-COMMAND-PATH`

Status: `CLOSED_NON_BLOCKING`

Class: verifier/process, not repo behavior failure.

Owner/source:

- Fresh subagent rerun command evidence and this closure ledger.

Release impact:

- Non-blocking after clean POSIX-path serial reruns and package validation, but
  the rerun abnormality had to be recorded before using the adversarial rerun as
  release proof.

Trigger:

- The adversarial rerun invoked Bash tests with Windows backslash paths, such as
  `bash tests\*.test.sh`, which failed because they are invalid Bash paths.
- The read-only rerun also had one broad PowerShell readback attempt with
  malformed quoting; the exact smaller readback was rerun successfully.

Affected commands/processes:

- `bash tests\*.test.sh` style invocations in the adversarial rerun.
- Broad PowerShell `rg` readback attempt in the read-only rerun.

Immediate containment:

- Stopped relying on the failed or malformed command output.
- Reran the affected evidence path with POSIX Bash paths and exact readback
  commands.
- Counted only successful reruns as release evidence.

Fix / containment type:

- Process containment and evidence substitution, not a repo behavior patch.

Evidence:

- Failed summary: Windows-backslash Bash test invocations failed in the
  adversarial rerun.
- Successful POSIX serial reruns:
  - `bash tests/repo-state.test.sh` -> pass
  - `bash tests/audit-spec.test.sh` -> pass
  - `bash tests/marker-order.test.sh` -> pass
  - `bash tests/release-asset.test.sh` -> pass
  - `bash tests/routing.test.sh` -> pass
  - `bash tests/install-copy-smoke.test.sh` -> pass
  - `bash tests/added-lines-clean.test.sh` -> pass
- `bash scripts/verify-package.sh` -> pass.

5 Whys:

1. Why was rerun evidence interrupted? Because Bash tests were invoked with
   Windows backslash paths and one broad readback had malformed quoting.
2. Why did that matter? Because invalid command syntax can look like a test
   failure or obscure whether the repo behavior was tested.
3. Why was false evidence possible? Because a failed command invocation could
   be normalized away after a clean rerun unless the abnormal path is recorded.
4. Why is this not a repo behavior failure? Because the same tests passed with
   valid POSIX paths, exact readback passed, and package validation passed.
5. Why is release safe to continue after correction? Because the abnormal
   command path is recorded, failed output is not counted as pass evidence, and
   clean rerun/package evidence replaces it.

Hansei:

- Gap: a verifier rerun introduced an invalid command path while testing
  release evidence.
- Cause: Windows path style leaked into Bash command invocation.
- Countermeasure: release-gate Bash commands must use POSIX paths; if path
  translation or quoting fails, record an Andon and rerun cleanly before using
  the evidence.
- Follow-up evidence: this ledger entry plus successful POSIX reruns and
  package validation.

Closure proof:

- The failed backslash-path and malformed-readback outputs are not counted as
  pass evidence.
- All affected tests passed with valid POSIX-path reruns.
- `bash scripts/verify-package.sh` passed after the reruns.
- No tag, GitHub Release, asset upload, or checksum/provenance publication
  happened before this abnormality was recorded.

Remaining risk:

- Windows/POSIX command translation mistakes may recur. They are not
  release-blocking unless required checks cannot pass cleanly with valid
  commands.

  are recorded with the primary Andon.
- No tag, GitHub Release, asset upload, or checksum/provenance publication
  happened before this record.

Countermeasure / Kaizen:

- If a release-gate command fails, hangs, times out, or is replaced by a
  rerun/substitute path, record an Andon first; then close it as blocking or
  non-blocking based on clean evidence.

Remaining risk:

- Future agents may still be tempted to summarize contained infra failures as
  harmless. The durable countermeasure is to preserve the abnormal path in the
  release ledger before claiming the gate is clean.
