# IMPLEMENTAUDIT v0.3.2.0 — stronger evidence, safer resumes, and executable audit plans

v0.3.2.0 is a substantial runtime-hardening release.

Where v0.3.1.0 primarily improved skill packaging, repository organization, read-only planning, and deterministic validation, v0.3.2.0 changes how IMPLEMENTAUDIT decides what work is warranted, what evidence counts, how failures and interruptions are handled, how plans and handoffs are reviewed, and when completion may honestly be claimed.

## What’s new

### Evidence is bound to the claim it proves

Required checks can now identify whether they test a structural, behavioral, or provenance property.
Evidence may be anchored to the exact repository revision that produced it, and closure claims must identify the surface actually verified—such as source, generated output, package, installed copy, running system, deployment, API, or publication.

A lower-layer success can no longer be promoted into proof of a higher layer.
Public proof claims can also carry an explicit PL1–PL7 proof level, separating prose presence, runtime requirements, structural validation, fixture demonstration, observed behavior, and fresh-executor proof.

### Failure handling now tracks origin and residual risk

The Andon system adds explicit handling for infrastructure failures, correct work placed in the wrong copy or layer, and false closure.

Long-running commands now use recorded launch intent, append-only status, explicit completion markers, owned-process containment, and states such as `terminal-state-unverified` and `infrastructure-failed`.
A missing completion marker is not silently converted into a product failure.

Occurrence state, audit completion, and remaining-risk disposition are kept separate, allowing safe containment and partial resolution without pretending that the entire case either failed or closed.

Repeated defects can also trigger review of the governing rule itself—including checks that passed only by luck or were scoped too narrowly.

### Resumed runs reread live state before acting

After a genuine context boundary—such as compaction, a new session, or a handoff resume—IMPLEMENTAUDIT now rereads the live run state before repository mutation.

It records the boundary, classifies remembered instructions by current applicability, refuses to replay already-satisfied one-time instructions, preserves standing constraints and authorizations, and continues from the current live action rather than restarting from a stale summary.

### Ordinary requests trigger the warranted audit depth

Users do not need a special “deep mode” keyword.

IMPLEMENTAUDIT derives the warranted audit actions from scope, uncertainty, risk, dependencies, evidence gaps, authorization state, and the intended executor.
Both selected actions and important omissions are recorded with reasons.

Materially broad audits must perform actual bounded specialist passes—parallel when supported, serialized when necessary.
A coverage table documents those passes but cannot substitute for them.

### Plans are designed for a cold executor

New phase and handoff specifications require ordered steps, exact file or symbol targets, per-step verification, expected success shapes, explicit scope boundaries, and plan-specific STOP conditions.

Executor-facing artifacts also receive an independent fresh-context cold review before preflight, dispatch, or handoff.
Self-critique remains required, but it is no longer treated as independent review.

### Handoffs are verified against live state

A receiving run now inspects incoming handoff packets rather than trusting their formal appearance.

Mechanically recomputable claims are re-derived, evidence references are rebound to the live repository, and authorization statements are preserved verbatim. Contradictions block only the work that depends on them.

## 4. Independent B3-v1 re-derivation and defect classification

The re-derivation imported the scorer whose bytes are identical to the tagged
scorer, parsed each bound `events.jsonl` as structured JSON events, and called
`score_events(fixture, events, {})`. Results:

- 12/12 manifest event hashes match the raw event bytes;
- 12/12 stored verdicts and status-ledger rows contain an empty property map;
- 0/12 independently rescored missions have all five properties true;
- each six-mission wave has five distinct property signatures, not one
  identical ceiling signature;
- the v1 reports' “6/6 all-scored-properties TRUE”, “IDENTICAL ceiling”, and
  “both arms score at ceiling on all five properties” statements are therefore
  unsupported by the cited verdicts and contradicted by the bound events.

Root-cause trace:

1. `eval/runner.py::score_bundle` recomputed repository changes.
2. When any changed path was outside `fixture.allowed_paths`, it immediately
   built and wrote a host-failure verdict.
3. That early verdict did not call `scoring.score_events` and serialized
   `properties: {}`.
4. B3-v1 deliberately authorized no mutation path while asking the mission to
   continue ANDON 251, so all seeded runs reached that early gate.
5. The release write-up interpreted narration/transcript observations as a
   complete property ceiling without a stored, reconstructible property
   matrix.

Confirmed classes/layers:

- `evidence-mismatch`: a specific ceiling claim cites verdicts that contain no
  property results and raw events that rescore below ceiling;
- `false-closure`: issue #35, issue #10, and the public release were closed on
  that unsupported acceptance claim;
- fixture/authorization defect: continuation was coupled to a forbidden host
  mutation and capsule evidence was not scored;
- verdict-composition and host-short-circuit defect: host safety erased product
  measurements;
- report-adjudication defect: unreconstructible narrative was promoted to a
  release-gate conclusion.

The direct event scorer is not presently classified as the causal defect: it
produces the missing measurements when called. Product behavior remains
unverified until B3-v2 supplies a coherent, pre-registered decision path.
