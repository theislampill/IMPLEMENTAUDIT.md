## 2. Canonical Integrated Runtime Flow

1. Safety read / instruction hierarchy: `skills/SKILL.md` stage 0 and
   `AGENTS.md` set authorization gates, repo instruction precedence, and
   forbidden operations.
2. Repo-content-as-data and trust-boundary handling: `audit-category-matrix.md`
   and `PROTOCOL.md` classify audited source, diffs, examples, fixtures, issue
   text, PR text, docs snippets, web/tool output, and donor content as data
   unless admitted by the instruction hierarchy.
3. Input gate and audit-object creation: `routing.md` converts direct,
   embedded, goal-synthesis, or casual-build input into a `tdqyq-audit-object`.
4. Invocation-shape classification: `skills/SKILL.md` and `AGENTS.md` define
   embedded governance, direct governance, goal synthesis, and governed
   casual-build intake.
5. Greenfield / brownfield / mixed route: `routing.md` defaults existing repos
   to brownfield, classifies new governed artifacts as greenfield, and requires
   brownfield inspection before any greenfield-in-brownfield work.
6. DMAIC / DMADV route selection: brownfield repair goes through DMAIC;
   greenfield/replacement/direction work goes through DMADV; mixed work uses a
   DMAIC outer shell plus DMADV inner design.
7. Stage 2 Gemba / repo-map / optional Graphify orientation: live files and
   smokes remain proof; Graphify can orient owner/source discovery when present,
   fresh, and authorized.
8. Stage 3 risk / category / terminology / security / deep pressure:
   `audit-category-matrix.md`, `phase-design.md`, and
   `terminology-integration.md` attach findings to native routes, owner/source,
   evidence boundaries, risk, security pressure, and Andon triggers.
9. Stage 4 phase decomposition: phase specs must preserve owner/source,
   acceptance, STOP/Andon conditions, evidence strategy, rollback, and final
   audit coverage.
10. Stage 5 run-root artifacts: `.IMPLEMENTAUDIT/runs/<slug>/` artifacts hold
    ROADMAP, STATE, THINKING, PROTOCOL, sidecars, tools, context, and phase
    specs.
11. Stage 6 plan review / self-critique: plan review checks scope, drift,
    done criteria, owner/source, evidence, rollback, and execution boundaries.
12. Stage 6.5 preflight / Smoke A: baseline evidence is captured before
    mutation and classified as target, unrelated, or unclear.
13. Implementation / owner-source patching: patches must land at the
    owner/source surface, preserve scope, and respect generated-source,
    package, and authorization boundaries.
14. Smoke B / regression detection: after-state checks compare against Smoke A;
    regressions enter the regression protocol.
15. Andon / Jidoka escalation if abnormal: failed evidence, unclear owner,
    policy conflict, substituted command, sidecar staleness, impossible
    criteria, or regression stops the line without try caps.
16. Reconcile / Plan Closure / Control: every item closes as done, changed,
    blocked, deferred, or unverified; recurring defects need Poka-yoke,
    Standard Work, Control Plan, or explicit defer/handoff.
17. ActiveGraph or Markdown custody: custody is recorded through ActiveGraph
    when available/authorized or Markdown fallback otherwise; custody is not
    correctness proof.
18. Final audit: final verification proves terminal closure, boundaries, and
    remaining risks.
19. `AUDIT_COMPLETE` before `IMPLEMENTAUDIT_RUN_COMPLETE`: marker order and
    closure semantics are enforced by protocol and checker coverage.
