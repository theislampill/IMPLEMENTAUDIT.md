## Andon / Hansei / 5 Whys

### ANDON-20260607-V025-BASH-PARALLEL-HICCUP

Status: `CLOSED_NON_BLOCKING`

Class: infra/process, not repo behavior failure

Trigger: focused checks attempted through the parallel shell wrapper timed out
or returned Bash service errors. Affected commands included
`tests/continuity.test.sh`, `tests/phase-validation.test.sh`,
`tests/sidecars.test.sh`, `scripts/check-planner-stages.sh`, and
`tests/release-asset.test.sh`.

Containment: stopped relying on the parallel wrapper output, reran affected
checks serially in fresh shell invocations, and counted only successful serial
reruns plus the final full verifier as release evidence.

Evidence: serial reruns passed for `tests/claim-run.test.sh`,
`tests/continuity.test.sh`, `tests/phase-validation.test.sh`,
`tests/sidecars.test.sh`, `tests/capability-ledger.test.sh`,
`scripts/check-planner-stages.sh`, `scripts/check-routing.sh`, and
`tests/release-asset.test.sh`; `bash scripts/verify-package.sh` passed after
the containment path.

5 Whys:

1. Why was release evidence interrupted? Because focused test execution through
   the parallel wrapper hit shell-service errors or timeout behavior.
2. Why did that matter? Because release evidence could be falsely counted from
   a failed or incomplete command path.
3. Why was false evidence possible? Because wrapper failure can obscure whether
   a test failed, hung, or the shell itself failed.
4. Why is this not a repo behavior failure? Because the same affected checks
   passed serially in fresh shell invocations and the full package verifier
   passed.
5. Why is it safe to continue? Because the abnormal path is recorded, failed
   wrapper output is non-evidence, and clean serial reruns replaced it.

Hansei: transient validation infrastructure trouble must still be recorded as
an abnormal release-gate path before being closed as non-blocking.

Countermeasure / Kaizen: if a required gate fails, hangs, times out, shell
errors, or is replaced by a rerun/substitute path, record an Andon first and
then close it only with clean bounded evidence.

### ANDON-20260607-V025-AUDIT-COMMAND-SUBSTITUTIONS

Status: `CLOSED_NON_BLOCKING`

Class: process/evidence hygiene, not repo behavior failure

Trigger: an initial package-manifest command used an unavailable Python command
inside Git Bash, and the first external hygiene scan script had a local type
error. Both commands were malformed audit commands, not product checks.

Containment: treated both failed commands as non-evidence, reran package
manifest inspection through the Windows Python launcher, reran the external
hygiene scan with corrected local script logic, and recorded only the corrected
passes as evidence.

Evidence: corrected package manifest review reported 23 files under
`.claude-plugin/` and `skills/` only. Corrected external hygiene scan passed for
tracked source and the rebuilt `dist/IMPLEMENTAUDIT.skill` asset. Full
`bash scripts/verify-package.sh` passed.

5 Whys:

1. Why did audit evidence need substitution? Because two local audit helper
   commands were malformed for the active shell/runtime.
2. Why did that matter? Because a failed audit helper could be mistaken for a
   product failure or, worse, a partial check could be counted as proof.
3. Why was that possible? Because one-off release audit commands sit outside
   repo-owned validation and have less fixture coverage.
4. Why is this non-blocking? Because the failed helper commands were replaced
   by corrected bounded checks, and the repo-owned package verifier passed.
5. Why is release still safe after correction? Because only corrected checks are
   counted, the abnormal path is recorded, and the underlying package/source
   evidence is clean.

Hansei: one-off audit commands must be treated with the same evidence
discipline as repo checkers: failed helper paths are abnormal and cannot be
quietly converted into proof.

Countermeasure / Kaizen: final audit may use external one-shot commands, but a
failed helper command must be classified as non-evidence and rerun with clean
bounded output before any closure claim.
