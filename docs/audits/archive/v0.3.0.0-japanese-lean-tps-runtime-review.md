# 日本語Lean/TPSレビュー

## 1. 総合判定

条件付き合格

Lean/TPS語彙をLLMエージェント実行系へ適用するのは通常の製造現場とは異なるが、このリポジトリでは多くの語が実行ゲート、テンプレート、チェッカー、fixture、最終監査条件に接続されている。単なる借用語彙ではない。ただし、`TERMINOLOGY_INTEGRATED` 相当の完了宣言はまだ早い。用語統合の監査台帳が `Status: in progress`、`Final verification: Pending` のままであるため、完了フラグはブロックすべきである（`C:/workspace/ai/improveimplementaudit/IMPLEMENTAUDIT/docs/audits/v0.3.0.0-runtime-terminology-proof-audit.md:5`, `:149-151`）。

## 2. 良い点

- 用語の所有境界が明確である。`terminology-integration.md` は第二用語集ではなく、既存参照への薄い優先順位表として定義されている（`C:/workspace/ai/improveimplementaudit/IMPLEMENTAUDIT/skills/references/terminology-integration.md:3-17`）。
- Lean用語は実行時のふるまい、テンプレート項目、チェッカー、最終監査規則に接続される、と明記されている（`C:/workspace/ai/improveimplementaudit/IMPLEMENTAUDIT/skills/references/lean-operating-discipline.md:3-5`）。
- Jidoka / Andon は停止、原因把握、対策、再実行、handoff 条件まで含む実行ループになっている（`C:/workspace/ai/improveimplementaudit/IMPLEMENTAUDIT/skills/templates/PROTOCOL.md:352-378`, `:387-394`）。
- DMAIC / DMADV の brownfield / greenfield ルーティングは適切に分離されている（`C:/workspace/ai/improveimplementaudit/IMPLEMENTAUDIT/skills/references/routing.md:159-203`）。
- Poka-yoke は具体的チェッカー群に接続されている（`C:/workspace/ai/improveimplementaudit/IMPLEMENTAUDIT/skills/references/lean-operating-discipline.md:26`）。
- docs portal は plain-first を採用し、用語を権威化しない方針が明確である（`C:/workspace/ai/improveimplementaudit/IMPLEMENTAUDIT/docs/portal/pages/terminology.html:2-5`, `:40-54`）。

## 3. 重大な懸念

- P0: 用語統合の proof audit が未完了である。`Status: in progress`、`Final verification: Pending` のままなので、`TERMINOLOGY_INTEGRATED` を完了扱いにしてはいけない（`docs/audits/v0.3.0.0-runtime-terminology-proof-audit.md:5-8`, `:149-151`）。
- P1: Hoshin Kanri / 方針管理は全文検索で検出されず、現状は active runtime vocabulary ではない。`strategy alignment` と `top objective` は存在するが、方針管理としての catchball、年度方針、方策展開、レビュー周期、責任展開は定義されていない（`skills/references/planning-depth.md:39-41`, `skills/templates/THINKING.md:9-13`）。導入するより plain “strategy alignment” のままにするべきである。
- P1: Plan Closure は一部文書では終端ステータスだけに見える。`SKILL.md` は `done/changed/blocked/deferred/unverified` に閉じることを定義するが、再発防止は `lean-operating-discipline.md` と `plan-lifecycle.md` 側に分散している（`skills/SKILL.md:301-311`, `skills/references/lean-operating-discipline.md:31`, `skills/references/plan-lifecycle.md:30-32`）。新規保守者には Control との接続を前面化した方がよい。
- P2: `AGENTS.md` は権威入口として有用だが、1274行規模で、用語説明・履歴・anti-repeat・参照目録が混在している。重複権威というより、読み手負荷と stale claim のリスクがある（`AGENTS.md:3`, `:983-1000`, `:1250-1268`）。

## 5. ドキュメント構造レビュー

- README は「Quick Vocabulary, Not Authority」と明記しており、重複権威化を避けている（`README.md:141-146`）。これは良い。
- docs portal も “not a glossary or authority layer” とし、plain-first を徹底している（`docs/portal/pages/terminology.html:2-5`）。新規利用者向けには適切。
- `terminology-integration.md` は薄い precedence contract として保たれている（`skills/references/terminology-integration.md:3-5`, `:36-68`）。ここは現在の構造の中核であり、維持すべき。
- `AGENTS.md` は「Authoritative project doc」として始まり、同時に runtime authority を references に委譲している（`AGENTS.md:3`, `:1250-1268`）。矛盾ではないが、初見エージェントが AGENTS の履歴規則を runtime authority と誤読するリスクがある。
- docs/audit-trail は履歴を現在証明と混同しないよう明記している（`docs/portal/pages/audit-trail.html:3-10`, `:85-96`）。これは品質保証上よい。
- ただし、用語統合 proof audit が未完了の状態を残しているため、README/CHANGELOG が “local package-validation target” と境界を置いていても、完了宣言だけは避けるべきである（`CHANGELOG.md:43-54`）。

## 6. KISS / 過剰形式化レビュー

- Hoshin Kanri / 方針管理は追加不要。現状の “strategy alignment” と “top objective” で十分であり、方針管理と呼ぶと過剰形式化になる。
- 初回オンボーディングでは Jidoka / Hansei / 5 Whys より「失敗した証拠で止める」「原因と対策を残す」を先に出すべき。portal はこの方向で良い（`docs/portal/pages/plain-english.html:52-62`）。
- `AGENTS.md` 冒頭の Operating vocabulary は短くて良いが、Kaizen/Hansei/Nemawashi だけが先に出るため、Lean語彙の印象が強い（`AGENTS.md:29-39`）。plain behavior first に寄せてもよい。
- `terminology-integration.md` は現在88行で薄いが、候補語が増えると再び第二用語集化しやすい。既に checker が120行上限と term-entry drift を見ているのは妥当（`scripts/check-terminology-integration.sh:62-87`）。
- Control Plan は有用だが、重い品質管理計画として展開せず、「どのチェッカー・テンプレート・CI・AGENTS規則で再発を防ぐか」に限定するのが適切。

## 8. 最終判定

現在の用語体系は「借り物を貼っただけ」ではなく、かなりの部分が実行制御として成立している。IMPLEMENTAUDIT は Lean/TPS語彙を保持してよい。

ただし、次の条件つきである。Hoshin Kanri / 方針管理は採用しない。現状の plain strategy alignment に留める。Plan Closure / Control の再発防止接続を `SKILL.md` 入口側にも明示する。`check-lean-discipline.sh` の意味検出を少し強める。そして何より、用語統合 proof audit が `Pending` の間は `TERMINOLOGY_INTEGRATED` を完了扱いにしない。
